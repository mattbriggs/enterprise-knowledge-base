# 6. Storage & Retrieval

Chapter 6 explains how the repository stores governed content, what it treats as authoritative, and which retrieval patterns follow from that choice. Storage is not a single layer in this system. The repository already uses multiple content forms, and each form exists for a different reason.

The current implementation has no live document database, graph database, or vector store. That is not a gap in explanation. It is part of the design boundary. The repository demonstrates the storage and retrieval model in file form so that the content contract remains inspectable and rebuildable from source.

## 6.1 Storage As A Set Of Contracts

The system currently works with four practical content forms:

- the authored Markdown source document
- the normalized chunk record
- the graph-oriented projection
- the published static artifact

These forms are not interchangeable. Each one answers a different engineering need.

### 6.1.1 Source Document

The source document is optimized for authorship, readability, and version control. It preserves the text in the form most natural to a human contributor.

### 6.1.2 Normalized Record

The normalized record is optimized for deterministic downstream processing. It is the first storage form that is explicitly machine-governed. In the current repository this is the most important derived artifact because it is the boundary that later graph, search, and publication work should consume.

### 6.1.3 Graph Projection

The graph projection is optimized for relationship-aware traversal. It does not preserve the entire authored document contract; instead, it preserves entities and relationships useful for graph loading and query.

### 6.1.4 Published Artifact

The published artifact is optimized for readers and delivery. It preserves the narrative surface of the content while relying on earlier transformations for structure and consistency.

## 6.2 What Counts As The Source Of Truth

The repository already implies a source-of-truth policy even though it does not yet implement a production database stack.

The authored Markdown file is the authoritative source for human-authored content. The normalized chunk record is the authoritative machine boundary for derived automation. The graph projection is not a peer to either of those forms. It is a downstream representation whose job is to support graph-oriented work.

This distinction prevents a common failure mode in knowledge systems: allowing multiple derived stores to drift into pseudo-authoritative status. Once that happens, the system loses a clean answer to the question, "Where does this fact get corrected?"

The operating rule is simple:

- correct content at the source document layer
- regenerate normalized records
- rebuild derived forms from those normalized records

That rebuild discipline is more important than a more elaborate storage stack would be at this stage.

## 6.3 Current Storage Surfaces In The Repository

The repository currently stores or emits the following practical artifacts:

- `content/*.md` as the author-facing source corpus
- `build/chunks.json` as a normalized chunk corpus generated by the ETL CLI
- `build/graph.json` as a graph-ready projection derived from the chunk corpus
- `site/` and export artifacts as publication outputs

Only the first of these is hand-authored. The others should be treated as deterministic derived forms.

## 6.4 Normalized Chunk Storage

The chunk corpus is the repository's clearest example of a machine-friendly storage contract. A chunk record currently contains:

| Field | Role In Retrieval And Storage |
| --- | --- |
| `id` | stable identity across derived forms |
| `heading` | local semantic label for the chunk |
| `content` | raw chunk text used for later indexing or rendering |
| `order` | sequence within the source document |
| `source_file` | provenance for rebuild and debugging |
| `source_path` | extraction-time location |
| `document_title` | document-level display label |
| `document_slug` | document-level grouping key |
| `metadata` | preserved source metadata for later consumers |

That structure supports several important operations:

- regrouping chunks by source document
- replaying the ETL pipeline deterministically
- projecting content into graph form
- later building lexical or semantic retrieval indexes
- checking provenance during validation and debugging

The record is intentionally flat except for `metadata`. That is a deliberate compromise. The top-level fields stay easy to query and reason about, while `metadata` preserves source detail without forcing premature normalization of every possible front matter key.

## 6.5 Retrieval Boundaries In The Current Repository

The repository currently supports three practical retrieval patterns.

### 6.5.1 Author-Facing Retrieval

Authors retrieve content by opening and editing the original Markdown files. This is the simplest retrieval path and remains the correct one for source edits.

### 6.5.2 Automation-Facing Retrieval

Automation retrieves content by reading the normalized chunk corpus. This is the correct retrieval path for any logic that needs stable identifiers, section ordering, or document-level grouping without reparsing the source corpus repeatedly.

### 6.5.3 Relationship-Facing Retrieval

Graph-oriented workflows retrieve content by reading the graph projection or by loading that projection into Neo4j and querying relationships there. This is the correct path for questions about chapter membership, tags, authorship, and graph traversal patterns.

The existence of these distinct retrieval paths is not redundancy. It is the point of the design.

## 6.6 Worked Example: From Source To Retrieval

Suppose a source document contains:

```md
---
title: Knowledge Handbook
slug: knowledge-handbook
author: Final State Press
tags:
  - etl
  - graph
---

## Normalization Boundary

The normalization boundary converts authored Markdown into stable records.

## Failure Modes

Malformed front matter should fail before publication.
```

After ETL, the chunk corpus supports retrieval like:

- "return the chunk with id `knowledge-handbook-001-normalization-boundary`"
- "return all chunks where `document_slug = knowledge-handbook` ordered by `order`"
- "return all chunks whose metadata contains the topic `graph`"

After graph projection, the same content supports relationship-oriented retrieval like:

- "which chapter contains this chunk?"
- "which topics are attached to this chunk?"
- "which chunks were written by this author?"

This is why normalized storage matters. It creates a cleaner retrieval substrate than raw Markdown alone can provide.

## 6.7 Current Limits Of Retrieval

The repository does not yet implement:

- full-text indexing
- semantic vector retrieval
- API-based retrieval
- incremental graph synchronization
- hybrid ranking across lexical and semantic signals

Those capabilities remain architectural targets rather than current features.

For now, retrieval is grounded in three concrete things: file access, normalized records, and graph projection.

## 6.8 Future Storage Layers

A fuller platform would likely introduce at least three explicit storage layers:

- a document store for structured content records
- a graph store for entities and relationships
- a search or vector index for retrieval-oriented access

Those future layers should still preserve the same boundary discipline already present in the repository:

- source content remains editable at the document layer
- normalization remains deterministic and replayable
- graph and retrieval stores remain derived consumers

The lesson is not "avoid multiple stores." It is "use multiple stores only when each has a clear contract and regeneration path."

## 6.9 Design Consequences

The storage model described here has several consequences for later implementation work.

First, any future database loader should consume normalized records rather than reparsing Markdown independently. Second, future retrieval layers should preserve the chunk identifier as a stable join key. Third, publication should remain rebuildable from source and normalized artifacts rather than from a graph store alone. Finally, any future synchronization logic should begin by protecting the source-of-truth boundary, not by adding convenience writes to derived stores.

Those consequences are what make this chapter part of the book's technical spine rather than merely a status note.
